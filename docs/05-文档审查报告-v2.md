## 🔴 Critical Issues（严重 — 必须在编码前修正）

### C1. 架构文档 §4 推理管线伪代码与 §0 双模型架构矛盾

**涉及文件：** [02-架构设计文档-v2.md](02-架构设计文档-v2.md) §0、§4

**问题描述：**

§0「模型架构总览」明确定义了 **三阶段管线**：
```
EfficientDet(检测框) → MobileNetV3-260(品类识别) → MobileNetV3-18(新鲜度分类)
```

但 §4 的伪代码只调用了一个 TFLite 分类器，且使用了 v1 的 API：

```kotlin
val logits = tfliteClassifier.classify(input.buffer)!!   // ❌ 未区分 260 类和 18 类
val result = modelMapper.mapToResult(logits, ...)         // ❌ v1 9 类映射，无法输出 260 类
```

**具体矛盾：**
1. `tfliteClassifier` 是 v1 的 18 类分类器，没有 260 类模型
2. `modelMapper.mapToResult()` 返回 v1 的 `FruitCategory` 枚举（9 种），无法映射 260 类
3. `DetectedItem` 构造时从 `result.fruitCategory.name` 取标签，只能得到 9 种 v1 类型

**对比：** [04-开发计划-v2.md](04-开发计划-v2.md) §P0-B 的伪代码反而是正确的——调用了 `classifier260` 和 `classifierFreshness` 两个分类器，使用 `labels_v2.txt` 中的 `isCookable` 字段。

**影响：** 如果开发以架构 §4 为准，会导致系统退化为 v1 的 9 类分类，260 类升级完全失效。

**修正方案：** 以开发计划 §P0-B 的伪代码为蓝本，重写架构 §4，明确区分三阶段管线。

### C2. 检测框坐标空间不匹配 Bug

**涉及文件：** [02-架构设计文档-v2.md](02-架构设计文档-v2.md) §4

**问题描述：**

```kotlin
val detectInput = ImagePreprocessor.resize(bitmap, 320, 320)   // ① 缩到 320×320
val (boxes, classes, scores) = detector.detect(detectInput)     // ② box 坐标在 320×320 空间
// ...
val crop = Bitmap.createBitmap(bitmap, box.x, box.y, box.w, box.h)  // ③ 在原始 bitmap 上裁剪！
```

EfficientDet 输入是 320×320，输出检测框坐标也是 320×320 空间。但裁剪时直接使用这些坐标去切原始 1920px 照片——**坐标空间完全错位**。

即使 `DetectedItem.bbox` 标注"归一化坐标 [0,1]"，伪代码中也完全没有归一化/反归一化步骤。

**正确流程应为：**
```kotlin
val detectInput = ImagePreprocessor.resize(bitmap, 320, 320)
val (boxes, classes, scores) = detector.detect(detectInput)

val items = validBoxes.map { box ->
    // 反归一化：320 空间 → 1920 空间
    val scaleX = bitmap.width / 320f
    val scaleY = bitmap.height / 320f
    val px = (box.x * scaleX).toInt()
    val py = (box.y * scaleY).toInt()
    val pw = (box.w * scaleX).toInt()
    val ph = (box.h * scaleY).toInt()
    val crop = Bitmap.createBitmap(bitmap, px, py, pw, ph)
    // ... 分类
}
```

**影响：** 所有检测框裁剪位置错误，导致后续分类模型输入错误区域，整个识别管线失败。

### C3. EfficientDet 对非 COCO 类别不可避免的漏检

**涉及文件：** [04-开发计划-v2.md](04-开发计划-v2.md) §P0-B, [02-架构设计文档-v2.md](02-架构设计文档-v2.md) §0

**问题描述：**

COCO 2017 的 80 个类别中，果蔬覆盖极其有限（仅 orange, apple, banana, broccoli, carrot, cucumber 等约 10 种）。以下 v2 中重要的可烹饪蔬菜在 COCO 中**完全不存在**：

- 辣椒、秋葵、苦瓜、茄子、西葫芦、花椰菜、芹菜、菠菜、四季豆、洋葱、姜、大蒜

EfficientDet-Lite0 以 COCO 预训练权重运行，对这些类别**不会输出检测框**，后续的双模型协同策略（分类兜底）从源头就不可行——检测阶段根本无框可兜。

**开发计划的矛盾：** §P0-B 提到"Synthetic Fruit Detection"用于微调，但在排期中完全没有体现（P0-B 仅排 1-2 天，且描述为"EfficientDet 检测集成"而非"微调"）。

**风险矩阵降级方案不成立：** 降级方案"整图直接送入分类模型"退化为单目标分类——输出一个全局分类结果，无法实现多目标检测框的核心需求。

**影响：** 大量蔬菜类别在检测阶段就丢失，多目标检测功能对蔬菜场景几乎不可用。

**修正方案：**
1. 在 P0-B 中增加 EfficientDet 在果蔬检测数据集上微调的环节（Roboflow Synthetic Fruit + 自标数据），排期至少 2-3 天
2. 或在降级方案中加入滑动窗口 + 分类的备选检测策略

### M4. 三模型加载策略缺失

**涉及文件：** [02-架构设计文档-v2.md](02-架构设计文档-v2.md) §0

**问题描述：**

三个 TFLite 模型总计磁盘 14MB，内存占用更关键（GPU delegate 下每个模型占用 10-20MB 显存/共享内存）。文档未定义：

1. 加载时机：Application.onCreate 全量加载（拖慢冷启动）还是按需懒加载（首次拍照等待更久）？
2. GPU delegate 分配策略：三个模型同时使用 GPU delegate 可能在低端设备（≤4GB RAM）上 OOM。是否需要某些模型降级为 CPU 推理？
3. 卸载策略：退出分析页后是否需要卸载模型释放内存？

**影响：** 低端设备上可能出现冷启动 ANR 或推理时 OOM 崩溃。

### M5. MediaPipe Tasks Vision 与 TFLite 依赖冲突风险

**涉及文件：** [02-架构设计文档-v2.md](02-架构设计文档-v2.md) §6 依赖变更表

**问题描述：**

新增的 `mediapipe-tasks-vision` 依赖内部捆绑了**自己的 TFLite runtime**（MediaPipe 使用 TFLite 作为推理后端）。现有项目直接依赖 `org.tensorflow:tensorflow-lite` 2.14.0。两个版本的 TFLite runtime 在同一 ClassLoader 下可能导致：

- 符号冲突（duplicate class）
- GPU delegate 版本不匹配导致 native crash

这是 TFLite/MediaPipe 集成中的已知痛点。

**影响：** 依赖集成阶段可能遇到编译冲突或运行时崩溃，需要额外排查时间。

**修正方案：** 在依赖变更表中增加一行提示，建议使用 MediaPipe 内置的 TFLite 而不是独立依赖；或在排期中预留 1 天依赖调试时间。

### m3. 菜谱引擎标签匹配断层

**涉及文件：** [02-架构设计文档-v2.md](02-架构设计文档-v2.md) §3.3

架构 §3.3 的匹配逻辑：

```kotlin
val ingredients = cookable.map { it.label }                    // "Tomato_Cherry_Red"
val matched = presetRecipes.filter { recipe ->
    recipe.matchIngredients.count { it in ingredients } > 0    // "番茄" in ["Tomato_Cherry_Red"] → false!
}
```

Fruits-360 标签是英文（如 `Tomato_Cherry_Red`），菜谱 JSON 的 `matchIngredients` 理应是中文（如 `"番茄"`）。直接字符串包含匹配 **永远失败**。

**需要：** 在 `labels_v2.txt` 中增加 `display_name` 到菜谱食材名的映射层；或在 `RecipeEngine` 中增加标签归一化步骤。

### 🔴 Critical Issues（已修正）

#### C1. AnalysisUiState 缺少 `isAnalyzing` 字段

**位置：** §4.1 AnalysisUiState 数据类定义 vs AnalysisViewModel.startAnalysis()

`AnalysisUiState` 数据类定义了 9 个字段，但 `startAnalysis()` 和错误处理中多处使用了 `isAnalyzing = false` 字段——该字段在数据类中不存在，会导致编译错误。

**修正：** 添加 `val isAnalyzing: Boolean = true` 字段（标注为 @deprecated，推荐使用 `screenState` 替代）。

#### C2. AnalysisViewModel 缺少 `@ApplicationContext` 注入

#### M7. RecipeEngine 评分示例表数学错误

**位置：** §6.3 匹配示例表

**修正：** 重写示例表，增加"凉拌黄瓜"完整匹配示例（score=1+3=4），修正所有数学和逻辑错误。

**位置：** §7.2 AnalysisBottomSheet

Material3 `ModalBottomSheet` 原生仅支持 `Hidden / PartiallyExpanded / Expanded` 三个 SheetValue。文档将 `PartiallyExpanded → COLLAPSED`、`Expanded → HALF` 映射后，**FULL 态无法通过 SheetValue 直接表示**。原文代码中 `else -> SheetState.FULL` 分支永远不会触发。
