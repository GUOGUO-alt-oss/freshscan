## 1. 严重问题（必须立即处理）

### 🔴 C1 — ProduceInfoEngine Hilt 重复绑定（潜在编译错误） ✅ 已修复

- **位置**: `ProduceInfoEngine.kt:21-26` + `AppModule.kt:148-154`
- **严重度**: Critical

`ProduceInfoEngine` 同时标注了 `@Singleton @Inject constructor`（自动 Hilt 构建）和在 `AppModule` 中的 `@Provides @Singleton` 手动提供。这与 v3 审查中 C2（DietPlanEngine 重复绑定）是完全相同的问题类型。虽然当前构建可以通过（Hilt 可能在某些条件下静默处理），但在 KSP 严格模式下或依赖变更后会触发 `[Dagger/DuplicateBindings]` 编译错误。

```kotlin
// ProduceInfoEngine.kt — 自动绑定
@Singleton
class ProduceInfoEngine @Inject constructor(
    @ApplicationContext private val context: Context,
    private val aiService: AIService,
    private val labelNormalizer: LabelNormalizer
)

// AppModule.kt:148-154 — 手动绑定（冲突！）
@Provides @Singleton
fun provideProduceInfoEngine(
    @ApplicationContext ctx: Context,
    aiService: AIService,
    labelNormalizer: LabelNormalizer
): ProduceInfoEngine = ProduceInfoEngine(ctx, aiService, labelNormalizer)
```

**修复方案**: 删除 `AppModule` 中的 `provideProduceInfoEngine()` 方法，保留 `@Inject constructor`（与 C2 修复方式一致）。

### 🔴 C2 — API Key 明文嵌入 Release APK BuildConfig

- **位置**: `build.gradle.kts:44,60`
- **严重度**: Critical

debug 和 release 构建都将 API Key 以 `buildConfigField("String", "AI_API_KEY", ...)` 方式嵌入 BuildConfig。release APK 中此字符串常量即使经过 R8 混淆也可通过反编译 trivially 提取（`BuildConfig` 类的静态字段不会被混淆）。任何下载安装 APK 的人都可以获取 DashScope API Key。

```kotlin
// build.gradle.kts:44 (debug)
buildConfigField("String", "AI_API_KEY", "\"${localProperties.getProperty("AI_API_KEY", "")}\"")
// build.gradle.kts:60 (release)
buildConfigField("String", "AI_API_KEY", "\"${releaseKey}\"")
```

**修复方案**: 
1. 短期：release buildType 使用不同的受限 API Key（配额限制 + 域名绑定）
2. 长期：将 API Key 移至后端代理，或运行时从 Android Keystore / 加密 SharedPreferences 读取
3. 至少：debug 和 release 使用不同 Key，release Key 设置严格的 DashScope 配额

## 2. 高优先级问题（本迭代内修复）

### 🟠 H1 — extractValue() 双 substringAfter 截断含冒号的值 ✅ 已修复

- **位置**: `DietPlanEngine.kt:257`
- **严重度**: High

`extractValue()` 函数链式调用 `.substringAfter("：").substringAfter(":")` 提取标签后的值。当 AI 输出的值本身包含半角冒号时（如 `"菜名：经典番茄炒蛋（妈妈的味道：改良版）"`），第二个 `substringAfter(":")` 会截断冒号前的内容，导致菜名变为 `"改良版）"`。

```kotlin
val afterLabel = line.substring(idx + label.length)
return afterLabel.substringAfter("：").substringAfter(":").trim()
// 输入: "：value:with:colons"
// substringAfter("：") → "value:with:colons"
// substringAfter(":")  → "with:colons"  ← BUG: 截断了 "value"
```

**修复方案**: 使用 `removePrefix` 替代链式 `substringAfter`：
```kotlin
return afterLabel.removePrefix("：").removePrefix(":").trim()
```

### 🟠 H2 — AnalysisViewModel.loadProduceInfo 快速点击竞态 ✅ 已修复

- **位置**: `AnalysisViewModel.kt:315-334`
- **严重度**: High

用户快速点击检测项时，每次点击都 launch 新协程 collect `produceInfoEngine.getInfo()` 的 Flow。这些协程并发运行，慢的早期请求可能覆盖快的后期请求结果，导致用户看到错误的食材百科信息。

```kotlin
private fun loadProduceInfo(label: String) {
    viewModelScope.launch {  // 未取消前一个 job
        produceInfoEngine.getInfo(label).collect { info ->
            _selectedItemInfo.value = info  // 可能被旧协程覆盖
        }
    }
}
```

**修复方案**: 保存 Job 引用，新请求前取消旧 Job：
```kotlin
private var produceInfoJob: Job? = null

private fun loadProduceInfo(label: String) {
    produceInfoJob?.cancel()
    produceInfoJob = viewModelScope.launch { ... }
}
```

### 🟠 H3 — ProGuard 缺少 MediaPipe 规则（release 构建崩溃风险） ✅ 已修复

- **位置**: `proguard-rules.pro`
- **严重度**: High

项目依赖 `com.google.mediapipe:tasks-vision:0.10.14`（EfficientDetEngine 使用），但 ProGuard 规则中完全没有 MediaPipe 的 keep 规则。MediaPipe 内部使用反射加载 native 库和处理器类，R8 会将其移除，导致 release 构建在物体检测时崩溃（`ClassNotFoundException` / `NoSuchMethodError`）。

**修复方案**:
```proguard
# MediaPipe Tasks Vision
-keep class com.google.mediapipe.** { *; }
-dontwarn com.google.mediapipe.**
```

## 3. 中优先级问题（下迭代修复）

### 🟡 M1 — FormatUtil 共享 SimpleDateFormat 非线程安全 ✅ 已修复

- **位置**: `FormatUtil.kt:13-15`
- **严重度**: Medium

`SimpleDateFormat` 不是线程安全的。三个静态共享实例（`timeFormat`, `dateWithoutYearFormat`, `fullDateFormat`）如果被并发调用（如 LazyColumn 渲染多个历史项时），内部 Calendar 状态可能被破坏，产生乱码日期字符串或崩溃。

**修复方案**: 每次调用创建本地实例，或改用 `java.time.format.DateTimeFormatter`（线程安全，但需 minSdk 26，当前 minSdk 24 不可用）。推荐方案：

```kotlin
fun formatTimestamp(timestamp: Long): String {
    val timeFormat = SimpleDateFormat("HH:mm", Locale.getDefault())
    // ... 局部变量
}
```

### 🟡 M2 — TFLiteClassifier.classify() use-after-close 竞态 ✅ 已修复

- **位置**: `TFLiteClassifier.kt:77-113`
- **严重度**: Medium

`classify()` 在 synchronized 块内读取 `currentInterpreter` 引用后释放锁，然后在锁外调用 `currentInterpreter.run()`。此间隙中另一线程可能调用 `close()` 关闭 native 资源，导致 `run()` 操作在已关闭的 interpreter 上执行，抛出 `IllegalStateException` 或 native crash。

**修复方案**: 将 `run()` 调用移入 synchronized 块内。由于 TFLite 推理本身是单线程的，序列化调用不会有性能损失。

### 🟡 M3 — meal_history 表无上界/无修剪策略 ✅ 已修复

- **位置**: `MealHistoryDao.kt:11-12` + `DietPlanEngine.kt:30-50`
- **严重度**: Medium

与 `history` 表使用 `insertAndTrim()` 限制 50 条不同，`meal_history` 表无限增长。`MealHistoryDao.getAll()` 无 LIMIT 子句，去重查询 `mealHistoryDao.getAll().first()` 加载全表到内存只为取前 10 条。

**修复方案**:
1. 添加 LIMIT 查询：`@Query("SELECT title FROM meal_history ORDER BY generatedAt DESC LIMIT 10")`
2. 添加 `insertAndTrim()` 类似 history 表的实现
3. 在 insert 前清理超过 90 天的旧记录

### 🟡 M4 — Migration 3→4 使用 SELECT *（列顺序脆弱） ✅ 已修复

- **位置**: `HistoryDatabase.kt:156`
- **严重度**: Medium

`INSERT INTO user_profile_new SELECT * FROM user_profile WHERE id = 1` 依赖新旧表列顺序完全一致。如果 Room 生成的列顺序与手动 DDL 不同，数据将被静默插入错误列（如 age 值进入 heightCm 列）。

**修复方案**: 显式列出所有列名：
```kotlin
db.execSQL("""
    INSERT INTO user_profile_new (id, spiceLevel, saltLevel, ...)
    SELECT id, spiceLevel, saltLevel, ...
    FROM user_profile WHERE id = 1
""")
```

### 🟡 M5 — FavoriteRecipeDao.isFavorited() 同步方法无 suspend ✅ 已修复

- **位置**: `FavoriteRecipeDao.kt:21`
- **严重度**: Medium

`isFavorited(recipeId: String): Boolean` 是同步 DAO 查询（非 suspend、非 Flow）。Room 强制同步查询不能在主线程执行。如果任何调用者在主线程调用将抛出 `IllegalStateException`。

**修复方案**: 添加 `suspend` 或返回 `Flow<Boolean>`。

### 🟡 M6 — UriInputStreamProvider 领域层导入 Android 框架类 ✅ 已修复

- **位置**: `domain/common/UriInputStreamProvider.kt:3`
- **严重度**: Medium

`UriInputStreamProvider` 接口导入 `android.net.Uri`，违反 Clean Architecture 领域层纯 Kotlin 原则。v3 审查修复了 `RectF` 泄露（H5），但遗漏了此处的 `Uri` 泄露。

**修复方案**: 将 `Uri` 参数改为 `String` 类型，在实现层（`AndroidUriInputStreamProvider`）转换为 `android.net.Uri`。

### 🟡 M7 — ModelConfigV2 存在无用的未限定绑定 ✅ 已修复

### 14.1 Bug #1 — ModalBottomSheet 下滑后页面卡死、返回键无效

- **严重度**: High（阻塞用户操作）
- **位置**: `AnalysisScreen.kt:343`
- **现象**: 识别后底部卡片弹出，用户下滑卡片后卡片消失但页面不退出，左上角 TopAppBar 返回键也无法点击

**根因分析**:

v4.0 M8 修复将 `onDismissRequest` 从 `onRetake` 改为 `{ }`（空回调），防止下滑触发重拍。但空回调导致 Material 3 的 `ModalBottomSheet` 在 sheet 内容隐藏后，composable 仍留在组件树中，scrim（半透明遮罩层）继续拦截所有触摸事件——包括 TopAppBar 返回按钮的点击事件。

```kotlin
// 修复前 — scrim 层拦截所有触摸，返回键无响应
ModalBottomSheet(onDismissRequest = { }, ...)

// 修复后 — 下滑时导航回主页，scrim 随 composable 移除而消失
ModalBottomSheet(onDismissRequest = onDismiss, ...)
// 其中 onDismiss 绑定到 onNavigateBack（popBackStack 到 HOME）
```

**修复方式**: 给 `AnalysisBottomSheet` 新增 `onDismiss: () -> Unit` 参数，绑定到 `onNavigateBack`（`navController.popBackStack(Routes.HOME)`）。用户下滑卡片后，`onDismissRequest` 触发导航返回主页。

### 14.2 Bug #2 — 扫描食材后所有信息为空（简介、营养、保存方法全空）

- **严重度**: High（核心功能失效）
- **位置**: `AnalysisViewModel.kt:297`、`ProduceInfoEngine.kt:42-60`、`produce_info.json`、`LabelNormalizer.kt:44-52`
- **现象**: 识别出"土豆"后点击进入食材详情页，简介、营养成分、保存方法全部空白，营养值全为 0

**根因分析**: 两个问题叠加导致：

1. **本地数据库覆盖不足**: `produce_info.json` 仅包含 5 种食材（苹果、橙子、香蕉、番茄、黄瓜）。18 类新鲜度模型能识别的土豆、甜椒、苦瓜、秋葵等均不在库中，`getCoreInfo()` 返回空壳 `ProduceInfo`（所有字符串为空，营养值为 0）。

2. **v1 降级路径 label 映射断裂**: 当 260 类模型不可用时，pipeline 降级到 18 类模型，`DetectedItem.label` 被设为 `FruitCategory.POTATO.name.lowercase()` = `"potato"`。`LabelNormalizer.normalize("potato")` 在 `labels_v2_normalization.json`（仅有 `Potato_Red` 等 v2 标签）和 `ModelConfigV2.labels` 中均无法匹配，最终 `categoryName = "potato"`（非中文"土豆"），本地查找二次 miss。

```
[模型识别] → label="potato", displayName="土豆"
  ↓
[AnalysisViewModel.onItemClicked] → loadProduceInfo(item.label)  ← 传了英文 label
  ↓
[ProduceInfoEngine.getInfo("potato")] → normalize("potato") → "potato"（未映射到中文）
  ↓
[getCoreInfo("potato")] → 空壳 ProduceInfo（intro="", calories=0）
  ↓
[fetchAIExtension] → AI 服务失败 → Flow 仅发射空壳 → UI 显示全空
```

**修复方式**（6 项改进）:

| # | 改进 | 文件 | 说明 |
|---|------|------|------|
| 1 | 传 displayName 替代 label | `AnalysisViewModel.kt` | `onItemClicked(item.label)` → `onItemClicked(item.displayName)`，直接用中文名"土豆"查询 |
| 2 | retryAIExtension 同步 | `AnalysisViewModel.kt` | `loadProduceInfo(info.label)` → `loadProduceInfo(info.displayName)`，保持缓存 key 一致 |
| 3 | 扩充本地数据库 | `produce_info.json` | 从 5 种扩充到 14 种（新增土豆、甜椒、苦瓜、秋葵、葡萄、西瓜、草莓、茄子、胡萝卜），覆盖 18 类模型全部品类 |
| 4 | AI 扩展增强完整信息生成 | `ProduceInfoEngine.kt` | 当 core info 缺失时（intro 为空且 calories=0），AI 请求完整食材信息（简介+营养+保存+时令），并合并到 ProduceInfo |
| 5 | LabelNormalizer displayName 反向查找 | `LabelNormalizer.kt` | 新增第 3 层 fallback：当 v2 label 匹配失败时，搜索所有 v2 labels 的 displayName 字段进行反向映射 |
| 6 | ProduceInfoSheet 条件渲染 | `ProduceInfoSheet.kt` | 空内容段落（简介、营养、保存方法、时令）不再显示空白标题，提升无数据时的 UX |
