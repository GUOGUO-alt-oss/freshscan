### 0.2 双模型架构

```
拍照 → Bitmap
         │
    ┌────┴────┐
    │ 检测阶段  │  EfficientDet-Lite0 (COCO 预训练)
    │          │  → 输出: 每个果蔬的 bounding box
    └────┬────┘
         │ 对每个检测框
    ┌────┴────┐
    │ 识别阶段  │  MobileNetV3-Small (Fruits-360 260类 训练)
    │          │  → 输出: "红富士苹果" / "樱桃番茄" / "紫甘蓝" ...
    └────┬────┘
         │
    ┌────┴────┐
    │ 新鲜度   │  MobileNetV3-Small (v1 18类模型 复用)
    │ (辅助)   │  → 输出: 新鲜 / 腐烂 / 不确定
    └────┬────┘
         │
    DetectedItem { label, freshness, bbox, isCookable }
```

## P0-A：Fruits-360 分类模型训练

### 目标
用 Fruits-360 的 260 类数据训练 MobileNetV3-Small，替换当前 18 类模型。

### 训练脚本

```python
# training/fruits360/train.py
# 与当前 train.py 结构相同，参数调整：

NUM_CLASSES = 260          # 18 → 260
BATCH_SIZE = 64            # 增加（260 类需要更多 batch）
MAX_EPOCHS = 80            # 更多 epoch
EARLY_STOP_PATIENCE = 8
LEARNING_RATE = 5e-4       # 稍高（类多需要更快收敛）

# 数据: Fruits-360 Training (137K) / Test (45K)
# 模型: MobileNetV3-Small (ImageNet 预训练)
# 输入: [1, 224, 224, 3] float32 [0,1]
# 输出: [1, 260] float32 raw logits

# 训练时间: CPU ~12-18 小时 / GPU ~2-3 小时
```

### 标签文件

## P0-B：EfficientDet 检测集成

### 策略变更（v2.4 更新 — 增加微调环节）

原计划 COCO→果蔬手动映射（方案 A 缺陷：辣椒/秋葵/苦瓜检测不到）。

**新策略：双模型协同 + 果蔬检测微调**
1. EfficientDet-Lite0（COCO 预训练）→ 提供通用"物体"检测能力
2. **🆕 在 Synthetic Fruit Detection (Roboflow) 上微调 EfficientDet-Lite0**，增加果蔬特有的检测能力（辣椒、秋葵、苦瓜、西葫芦等 COCO 不含的类别）
3. 检测到的每个框 → MobileNetV3 Fruits-360 分类 → 精确果蔬名称
4. 如果 EfficientDet 漏检 → 降级为滑动窗口检测（Sliding Window, stride=112px, 224×224 crop）+ 分类模型兜底

**微调数据：**
| 数据集 | 图片数 | 用途 |
|--------|--------|------|
| Roboflow Synthetic Fruit | 6,000 | 果蔬检测微调（YOLO → TFLite 转换） |
| 自标数据（可选） | 200-500 | 补充中餐常见蔬菜（苦瓜、秋葵、茄子等） |

**P0-B 排期更新：** 1-2 天 → 3-4 天（增加微调训练 + 数据集转换）

### 实际流程

```kotlin
fun detect(bitmap: Bitmap): List<DetectedItem> {
    // Step 1: EfficientDet 找框
    val boxes = efficientDet.detect(bitmap, threshold = 0.4)
    
    // Step 2: 对每个框做 260 类识别
    return boxes.map { box ->
        val crop = bitmap.crop(box)
        val input = preprocessor.preprocess(crop)
        val result = classifier260.classify(input)   // Fruits-360 模型
        val freshness = classifierFreshness.classify(input)  // v1 模型
        
        DetectedItem(
            label = result.label,
            displayName = result.displayName,
            freshnessLevel = freshness.level,
            confidence = result.confidence,
            bbox = box,
            isCookable = result.isCookable  // 来自 labels_v2.txt
        )
    }
}
```

## 风险矩阵

| 风险 | 概率 | 影响 | 对策 |
|------|------|------|------|
| Fruits-360 训练 260 类精度不达标 (<85%) | 低 | 高 | 降级为 Top-50 常见品类；使用更大模型 MobileNetV3-Large |
| 260 类模型体积过大 (>6MB) | 中 | 低 | FP16 量化 → ~2.5MB |
| EfficientDet 漏检（COCO 不含蔬菜类） | 中 | 高 | Synthetic Fruit 微调 + 滑动窗口降级方案 |
| 菜谱 JSON 手工编写 100+ 道工作量大 | 中 | 低 | 先用 LLM 批量生成菜谱草稿 → 人工审核 |
| 双模型推理延迟过高 | 低 | 低 | 检测+分类+新鲜度串行 ~300ms；2s 粒子动画完全覆盖 |
| 🆕 MediaPipe/TFLite 依赖冲突 | 中 | 高 | 统一使用 MediaPipe 内置 TFLite runtime；预留 1 天依赖调试 |
| 🆕 ModalBottomSheet + LazyColumn 嵌套滚动冲突 | 高 | 中 | Compose 已知痛点，预留 1-2 天独立调试手势拦截 |
| 🆕 v1 新鲜度模型在裁剪框上精度下降 | 中 | 中 | 用 v1 测试集裁剪验证；如精度下降 >10% 则在裁剪框数据上微调 |
| 🆕 Process Death 恢复 | 中 | 高 | `SavedStateHandle` 保存 photoUri；分析页支持从 URI 恢复 |
| 🆕 isCookable 标注错误（260 行人工标注） | 中 | 低 | 二人交叉校验；单元测试覆盖 100% 标签加载 |
