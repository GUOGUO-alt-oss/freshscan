## 1. 严重问题（必须立即处理）

### 🔴 C1 — API 密钥提交到版本控制 ✅ 已修复

- **位置**: `gradle.properties:27`
- **严重度**: Critical

`gradle.properties` 中包含实际的 DashScope API 密钥 (`sk-ws-H.RPHPRLM...`)。文件注释写着 "never commit real keys"，但密钥确实存在。`.gitignore` 未排除此文件，密钥已进入 git 完整历史。

```properties
# gradle.properties:26-27
# AI_API_KEY should be in local.properties — never commit real keys
AI_API_KEY=sk-ws-H.RPHPRLM.2dQG...
```

**修复方案**:
1. 立即轮换 API 密钥
2. 将密钥移至 `local.properties`（已在 .gitignore 中）
3. `gradle.properties` 中仅保留空占位符或注释
4. 使用 BFG Repo-Cleaner 从 git 历史中清除密钥

### 🔴 C2 — DietPlanEngine Hilt 重复绑定（编译错误） ✅ 已修复

- **位置**: `DietPlanEngine.kt:16-20` + `AppModule.kt:158-163`
- **严重度**: Critical

`DietPlanEngine` 同时标注了 `@Singleton @Inject constructor`（自动 Hilt 构建）和在 `AppModule` 中的 `@Provides @Singleton` 手动提供。这会导致 Dagger/Hilt 编译报错 "bound multiple times"。

```kotlin
// DietPlanEngine.kt — 自动绑定
@Singleton
class DietPlanEngine @Inject constructor(
    private val aiService: AIService,
    // ...
)

// AppModule.kt — 手动绑定（冲突！）
@Provides @Singleton
fun provideDietPlanEngine(...): DietPlanEngine
```

**修复方案**: 删除 `AppModule` 中的 `@Provides` 方法，保留 `@Inject constructor`（更简洁）。

### 🔴 C3 — 缺少 INTERNET 权限 ✅ 已修复

- **位置**: `AndroidManifest.xml`
- **严重度**: Critical

清单文件注释称 "No network permission needed -- fully offline"，未声明 `INTERNET` 权限。但 v3 的 `QwenAIService` 通过 OkHttp 向 DashScope API 发起 HTTP 请求。没有此权限，所有 AI 功能（食材百科扩展、饮食计划生成）将在运行时失败。

**修复方案**:
```xml
<uses-permission android:name="android.permission.INTERNET" />
```
同时更新注释说明 v3 部分功能需要网络。

## 2. 高优先级问题（本迭代内修复）

### 🟠 H1 — EfficientDetEngine detect()/close() 非线程安全 ✅ 已修复

- **位置**: `EfficientDetEngine.kt:93-97, 105-109`

`ensureLoaded()` 已加 `synchronized(loadLock)`，但 `detect()` 读取 `objectDetector!!` 在锁之外，`close()` 也未同步。如果 `close()` 在另一个线程被调用，`objectDetector` 可能变为 null，导致 NPE 崩溃。

```kotlin
fun detect(bitmap: Bitmap): List<BoundingBox> {
    ensureLoaded()
    val result = objectDetector!!.detect(...)  // !! 在锁外，可能 NPE
}
```

**修复方案**: 将 `detect()` 和 `close()` 也纳入 `synchronized(loadLock)` 保护，与 `TFLiteClassifier` 模式一致。

### 🟠 H2 — ProduceInfoEngine.ensureCoreInfoLoaded() 竞态条件 ✅ 已修复

- **位置**: `ProduceInfoEngine.kt:69-71`

check-then-act 模式 `coreInfoCache ?: loadCoreInfo().also { ... }` 非原子操作。两个线程可能同时看到 null 并重复加载大型 JSON 资源。

**修复方案**: 使用 Kotlin `lazy { loadCoreInfo() }` 委托（默认线程安全）。同样的模式问题也存在于 `RecipeEngine.ensureLoaded()` 和 `LabelNormalizer.ensureLoaded()`，应一并修复。

### 🟠 H3 — DietPlanEngine parseDietPlan() 静默返回空计划 ✅ 已修复

- **位置**: `DietPlanEngine.kt:95-180`

JSON 解析失败时 catch 所有异常并返回 `DietPlan(dailyPlans = emptyList())`。该空计划仍被持久化到数据库，用户看到空白饮食计划且无任何错误提示。

**修复方案**: 抛出 `DietPlanParseException` 或返回 `Result<DietPlan>`，在 ViewModel 中展示有意义的错误。

### 🟠 H4 — PersonalizeViewModel.loadProfile() Flow 覆盖用户编辑 ✅ 已修复

- **位置**: `PersonalizeViewModel.kt:34-40`

`userProfileDao.get().collect` 是持续 Flow，每次数据库有新值都会覆盖 `_uiState.value`，包括用户正在编辑但未保存的修改。`isDirty` 标记和所有用户输入将丢失。

**修复方案**: 使用 `.first()` 替代 `.collect {}`，仅加载初始值一次。

### 🟠 H5 — 领域模型泄露 Android 框架类型 RectF ✅ 已修复

- **位置**: `DetectedItem.kt:5`

`DetectedItem.bbox` 使用 `android.graphics.RectF`，违反了 Clean Architecture 领域层纯 Kotlin 的原则。这使得纯 JVM 单元测试需要依赖 Android stubs。

**修复方案**: 创建 `data class BoundingBox(val left: Float, val top: Float, val right: Float, val bottom: Float)`，仅在 UI 层转换为 `RectF`。

### 🟠 H6 — Backup 规则未排除健康数据 ✅ 已修复

- **位置**: `backup_rules.xml`, `data_extraction_rules.xml`

两个文件均为空模板，而 `allowBackup="true"` 意味着所有应用数据（用户健康画像、饮食计划、过敏信息、身体指标）都会被备份。对于处理健康数据的应用，这是隐私风险。

**修复方案**: 添加排除规则：
```xml
<exclude domain="database" path="freshscan_db"/>
<exclude domain="datastore" path="taste_profile"/>
```

### 🟠 H7 — ProGuard 规则过于宽泛 ✅ 已修复

- **位置**: `proguard-rules.pro:29, 39-41`

`-keep class androidx.compose.** { *; }` 禁用了 Compose 所有 tree-shaking，显著膨胀 release APK。`-keep class com.example.freshscan.data.history.** { *; }` 阻止 Room 生成代码的优化。

**修复方案**: 移除 Compose 宽泛规则（Compose AAR 自带 consumer rules），Room 规则缩小到仅保留 Database 类和 Entity 注解。

### 🟠 H8 — QwenAIService 重试逻辑大量代码重复 ✅ 已修复

- **位置**: `QwenAIService.kt:115-121, 128-167`

`retryRequest()` 几乎是 `executeRequest()` 请求构建代码的完全复制。违反 DRY 原则，任何修改需在两处同步。

**修复方案**: 提取 `buildRequest()` 和 `executeOnce(request)` 公共方法。

## 3. 中优先级问题（下迭代修复 → 已修复）

### 🟡 M1 — TDEE 公式对 UNSPECIFIED 性别处理不当 ✅ 已修复

- **位置**: `DietPlanEngine.kt:82-87`

`calculateTDEE` 对非 MALE 的性别（包括 UNSPECIFIED）使用女性 BMR 公式（`-161`）。未设置性别的用户将获得偏低的卡路里计算。

**修复方案**: 对 UNSPECIFIED 使用平均值 `(male + female) / 2`，或提示用户填写。

### 🟡 M2 — PersonalizeViewModel.save() 无错误处理 ✅ 已修复

- **位置**: `PersonalizeViewModel.kt:119-131`

如果 `userProfileDao.upsert()` 抛出异常，`isSaving` 将永远为 true，UI 卡死。

**修复方案**: 包裹 try/catch/finally，在 finally 中重置 `isSaving`。

### 🟡 M3 — DietPlanViewModel addMealToShoppingList 无去重 ✅ 已修复

- **位置**: `DietPlanViewModel.kt:67-76`

每次点击都插入所有食材，不检查是否已存在。多次点击产生重复条目。

### 🟡 M4 — PersonalizeScreen savedSuccessfully 标记永不重置 ✅ 已修复
